01A

Member Hub

Logged-in player room, telemetry wall, and data surface map for the Bloomberg / JARVIS style dashboard vision.

Experience target

The long-term member experience is not a flat profile page. It is a room: one wall for generated island previews, another wall for player cards and earned status, and one main screen for a live telemetry deck with charts, heatmaps, ticker rails, and ecosystem cards.

  • Island wall: grid of generated island PNGs, recent runs, and named build folders.
  • Player wall: Epic identity, locker skin, room theme, tickets, platform role, and earned badges.
  • Main screen: large TV-style telemetry with stat cards, chart stacks, live rail updates, and cross-platform readiness indicators.
  • Room styling: premium workshop / hero-lab language rather than a generic account dashboard.

Data surfaces we can actually use

  • Epic login identity: account identity and session state from Epic authentication.
  • Platform data: islands, uploads, tickets, channels, announcements, room theme, admin state, and other data we store in Oracle.
  • Public Fortnite surfaces: public shop, cosmetics, and published-island analytics once a real island code is live.
  • Keyed Fortnite stat lookups: player stat snapshots when the external Fortnite stats key is configured.
  • Product-side EOS data: stats, achievements, and progression for products we own and wire ourselves, not arbitrary third-party Epic games.

⚠
Data boundary

Epic login does not equal universal access to every player save profile across Epic games. The correct model is: Epic identity + TriptokForge platform data + approved public Fortnite data + product-specific services we explicitly integrate.

Dashboard build order

  1. Stabilize the current member telemetry deck and keep all current cards sourced from real Oracle or Fortnite endpoints.
  2. Add an island gallery wall inside the member room using the player's generated previews and saved metadata.
  3. Add a player-card rack for badges, earned events, hub role, linked systems, and future TriptokForge-native achievements.
  4. Upgrade the center screen into an immersive chart deck: deltas, heatmaps, activity timeline, ranked ladders, and event cards.
  5. Add richer 3D or faux-3D panels only after the data rails are real and readable.

02

System Architecture

Oracle Always Free ARM running nginx, Gunicorn, Flask blueprints, Oracle DB, and OCI-backed media.

Infra map

  • VM: Oracle Always Free ARM on Ubuntu 22.04.
  • Proxy: nginx forwards public traffic to Gunicorn on 127.0.0.1:5000.
  • App: Flask blueprints deployed through systemd service islandforge.
  • Database: Oracle Autonomous with thin-mode python-oracledb.
  • Storage: OCI bucket triptokforge in us-ashburn-1.
  • Repo: github.com/iamjedi888/ver-perlinforge on branch main.

Blueprint structure

  • routes/platform.py handles home, forge, gallery, feed, community, admin, and health.
  • routes/api.py handles social APIs such as posts and likes.
  • routes/channels.py serves channels and channel suggestions.
  • routes/auth.py manages Epic OAuth setup and callback flow.
  • routes/whitepages.py serves the internal docs and track feed endpoints.

Deploy Guide

Push to GitHub main, pull on Oracle, restart the service, and verify health locally on the box.

01
Push the change

git add .
git commit -m "message"
git push origin main

02
Pull on Oracle

cd ~/ver-perlinforge/islandforge
git pull origin main

03
Restart and verify

sudo systemctl daemon-reload
sudo systemctl restart islandforge
sudo systemctl status islandforge --no-pager
curl http://127.0.0.1:5000/health

⚠
Deploy rule

Do not use git reset --hard on the Oracle instance for routine deploys. Treat the server like a pull target for origin/main.

03A

Repo Privacy

How to keep GitHub private while still allowing Oracle to deploy from main.

Recommended setup

  • Set the GitHub repository visibility to Private.
  • Keep your personal account as the only write-capable maintainer unless you intentionally add collaborators.
  • Give the Oracle VM a single read-only deploy key for this repository.
  • Use an SSH remote on Oracle instead of a public HTTPS clone.

Why this is the right model

  • Code stays private from everyone except approved GitHub accounts.
  • The Oracle box only gets the minimum read access needed to pull deploys.
  • You avoid storing personal GitHub credentials on the server.
🔒
Recommended control

Keep the site repo private and let Oracle pull with a read-only deploy key. Do not use a public repo just to make deploys easier.

Staff + Bot Operations

Role-aware ops model for admins, moderators, and bot operators, plus the curated LLM provider catalog used in the site console.

Role split

  • Admin: full operations control, including staff accounts, bot profiles, channels, broadcasts, announcements, and legacy admin modules.
  • Moderator: community-post moderation only. This role can edit or remove posts without touching channels, bot settings, or sitewide UI controls.
  • Bot Operator: can tune a linked bot profile such as ColorsTheForce without receiving the rest of the admin surface.

Ops surface

  • /ops is the role-aware login and dashboard for staff accounts.
  • The old /admin route remains the legacy full-control console and should stay admin-only.
  • Leave the username blank and use the root admin password only as a fallback, not as the long-term staff model.

Provider catalog design

  • Provider and model dropdowns are curated from official vendor documentation rather than scraped live at runtime.
  • This keeps the site stable, readable, and safe to edit from the admin surface.
  • Good current anchors for bot selection are Google Vertex AI, IBM watsonx.ai, NVIDIA NIM, Meta Llama, and Hugging Face provider routing.

Verse Architecture

Tarkov-Lite systems in Verse for UEFN using immutable player data and stateful UI overlays.

Core patterns

  • Immutable data: build fresh player records instead of mutating shared state.
  • UI state: store HUD references per player for reliable cleanup and redraw.
  • Material encoding: compress rarity and element into deterministic inventory slots.
  • Persistence: keep long-term player progression compatible with seasonal updates.
  • Death penalty: preserve risk and extraction tension as a core loop.
  • World layout: keep the 32 farm plots in one coastal social cluster near town, then push biome escalation outward.

Island Forge

Audio-to-heightmap and Verse package workflow for UEFN island generation.

01
Upload + analyze

Upload a track in Forge. The audio pass extracts sub-bass, bass, midrange, presence, brilliance, BPM, and duration, then maps those values into terrain signals.

02
Set scale + theme

Choose world size, water, wrap, farm direction, cluster density, and Fortnite chapter theme before you generate. Large worlds change the UEFN import procedure because World Partition becomes mandatory.

03
Generate run

Each run is saved locally as outputs/WorldName#N/ with heightmap.png, preview.png, layout.json, manifest.json, placement_plan.json, generated Verse data files, and a verse_package.zip when available.

04
Import to UEFN

Use the saved run folder as the handoff bundle. Import the heightmap, follow the World Partition note when the chosen size requires it, then place the generated Verse files and bind builtin Fortnite assets in-editor using the generated slot names and gallery hints.

Current target layout keeps all 32 farm plots in one shared coastal hub near town, with coherent Fortnite prefab kits assigned biome by biome instead of random prop mixing.

The package is now intended to be theme-aware and concrete: the selected Fortnite chapter changes biome material guidance, gallery recommendations, landmark inspirations, foliage slot bindings, and placement-plan output instead of only changing a label.

The important contract is now explicit: changing world scale changes the downstream UEFN steps. If Forge flags World Partition, the player needs to enable it before import instead of trying to brute-force the landscape into a default island.

⚠
Import rule

If the selected world scale exceeds roughly 2km, enable World Partition and Streaming in UEFN before importing the generated heightmap. The same warning is repeated in Forge, layout.json, and the generated package README.

Forge Runbook

  1. Open /forge, upload an audio track, and let the analysis complete so the terrain weights reflect the real file instead of defaults.
  2. Name the world and choose the Fortnite chapter theme you want the asset manifest to target.
  3. Set world scale, resolution, water level, wrap, farm direction, and cluster density. If the Forge World Partition flag appears, plan to enable World Partition in UEFN before import.
  4. Press Generate. Confirm the run saved into outputs/WorldName#N/ and download or open the generated files from the Forge locker.
  5. In UEFN, create or open the target island. For large worlds: Edit -> Project Settings -> World -> Enable World Partition, then World Settings -> Enable Streaming.
  6. Import heightmap.png as the landscape using the recommended resolution from Forge. Use preview.png as a visual reference and layout.json for metadata, town center, and plot positions.
  7. Copy the generated Verse files into your project or extract verse_package.zip. Keep the generated files together with asset_manifest.json, landscape_config.json, and placement_plan.json.
  8. Bind the generated Verse slots to builtin Fortnite gallery and prop assets inside UEFN. The generated package documents supported builtin content-browser hints; it does not require custom shipped assets.
  9. Treat the generated Verse files as data modules and binding scaffolds. The package tells UEFN what to place and what slots to wire; the editor still owns the final builtin asset binding and any project-specific runtime spawn logic.
  10. Launch a session, validate streaming cells, town placement, plot spacing, and coastline readability, then iterate by regenerating the next numbered run folder instead of overwriting the old one.

Tarkov-Lite Vision

Persistent extraction MMO inside Fortnite with risk, insurance, progression, and evolving world systems.

Players stage from a safe hub, run extraction zones for materials and kills, and decide whether to bank progress or press deeper for better loot.

Economy loop

  • Grow or gather materials, then sell or craft around rarity and element value.
  • Kill enemies to earn drops, XP, and quest progression.
  • Extract safely to keep the run. Die and the carry risk becomes real.
  • Use insurance tokens to protect the most important item in a run.
  • Level up into harder zones, more systems, and higher-end cosmetics.
15

Game Systems

Core gameplay systems that define the extraction loop and long-term progression.

Death penalty

  • Lower stacks create full-loss pressure and teach early caution.
  • Higher-stack players still take meaningful loss while keeping late-game progression viable.
  • Insurance creates decision tension instead of full safety.

Material economy

  • Five primary material families keep collection readable.
  • Eighteen elements and multiple rarity bands support broad crafting depth.
  • A deterministic slot and value model keeps UI and storage simpler.
